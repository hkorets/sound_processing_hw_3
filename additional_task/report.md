# Звіт додаткове завдання: Побудова пайплайну End-to-End AST

### 1. Огляд даних: основні особливості та проблеми
Для реалізації завдання ми створили синтетичний паралельний датасет, оскільки готові рішення для прямого перекладу з англійського аудіо в український текст (AST) є рідкісними.

* **Джерело:** Англомовні аудіозаписи (на основі бібліотеки `datasets`).
* **Метод анотування:** Використання LLM (OpenAI API) для автоматизованого перекладу англійських транскрипцій на українську мову.
* **Основні проблеми та виклики:**
    * **Обмежений обсяг:** Для прототипу ми використали всього 30 записів (Proof of Concept).
    * **Синхронізація модальностей:** Необхідність приведення аудіо-сигналу (Raw Audio) та тексту до єдиного формату, зрозумілого моделі (спектрограми + токени).

---

### 2. Опис використаних підходів
В основі архітектури лежить модель **Whisper-tiny** від OpenAI, адаптована за допомогою фреймворку **PyTorch Lightning**.

**Ключові етапи реалізації:**
1.  **Data Generation:** Створення пар `EN Audio -> UK Text` за допомогою запитів до LLM.
2.  **Preprocessing:** * Перетворення аудіо у 80-канальні **Мел-спектрограми** (Log-Mel Spectrograms).
    * Токенізація українських текстів за допомогою `WhisperTokenizer`.
3.  **Data Pipeline:** Написали `WhisperDataModule` та кастомний `DataCollator` для динамічного вирівнювання (padding) батчів.
4.  **Model Configuration:** Використали параметр `forced_decoder_ids` для примусового перемикання моделі в режим генерації українською мовою.
5.  **Training:** Налаштували тренувальний цикл з використанням AdamW оптимізатора та відстеженням функції втрат (Loss).

### 4. Гіпотези та думки щодо отриманих результатів
Під час інференсу було отримано результат: модель намагається відтворити звучання англійських слів українськими літерами (наприклад, *"Ін kwиод шегі"* замість *"Inquired shaggy"*).

**Аналіз результатів:**
* **Гіпотеза про "шорткати" (Underfitting):** Через критично малий обсяг даних (30 семплів) модель не змогла вивчити семантичний зв'язок (переклад). Вона знайшла найпростіший математичний шлях для зменшення Loss - кодування звуків схожими за звучанням символами іншого алфавіту.
* **Ефективність пайплайну:** Попри якість перекладу, технічний пайплайн працює: градієнти течуть, Loss падає, а механізм `forced_decoder_ids` змушує модель використовувати кирилицю.
* **Масштабованість:** Дана архітектура повністю готова до другого додаткового завдання (SOTA). Для отримання літературного перекладу необхідно масштабувати датасет до 1000+ прикладів.
